User profile redesign

Challenge

Business problem

As Lynk & Co was maturing and its number of customers was growing, there was a problem with supporting the users after the purchase. Users would have to call the Customer support for documents updates, damage report, managing their reservation, etc. This led to the company losing money and reputation. Also, it generated significant workload on Customer support specialist, which affected their job satisfaction and increased turnover.

User problem

For every simple action regarding their car or membership users, would have to call the Customer support center. The long waiting times, as well as the problems with explaining the issues over the phone, created a lot of discontent. Customers expected User profile to be the tool where they’d be able to manage their account and related services.

To address both problems, we needed to transition to a modular platform that would enable creating an automatic self-service tool. This tool would need to be easy to maintain and scale, and meet the users' rexpectations.

Findings

  • The company was failing to deliver on its “hassle-free” promise, which resulted in a frustrating user journey after the conversion.
  • The company was losing money on manually supporting the customers. Automation was needed.
  • The current version of the User profile section was hard-coded, and the dev team could not afford to constantly update the platform when new needs appeared.

Scope

The business needed a self-service platform that would reduce the waiting time for the users trying to receive support. Also, this would lift off the workload of the Customer service specialists.

Making the platform that could be built with blocks (instead of hard-coding it) would enable the dev team to focus on larger projects in the long run, leaving the maintenance and update tasks to site editors.

Arm wrestle with the stakeholders

We had an ongoing back and forth with the stakeholders during this process. One of the main concerns they had was the constant fear of the lack of capacity. On our side, we didn’t want to sacrifice the UX features. We tried to resolve this stand-off with planning.

To address their concerns, we did the following:

  • Broke the project down, focusing only on B2C customers first.
  • Planned gradual transition with 4 launches, where the 1st launch would be a transition from the current version to the modular one with the same features; the 4th launch would have 4 more new features: Appointments, Notifications, Support, Products & services.
  • Had regular check-ins with developers to make sure our approach to the design system was feasible and manageable.
  • Brought service owners into the conversation to better explain the business need for the features we had offered.

Design decisions

Improved navigation

To reduce task completion task, we redesigned the navigation based on the users expectations that were revealed during card sorting task. This way, we had only 3 (instead of 7) first level menu items and moved the rest of the menu items onto the second level. This way, users are met with only few options at a time.

UI update

To reduce the cognitive load, we also update the user interface design of the user profile. Mainly, we toned down the visual noise. Also, we used the same color scheme as on the brand side of the company's website.

Design system for modular approach

Since making a platform that could be built with blocks was one of the goals of this project, we have created a set of components for developers to build. Once built, these components would be used by site editors to build pages and make small updates to existing pages faster.

Result

This project resulted in signed-off high-fidelity mock-ups for a modular self-service platform that could be assembled using a set of proposed components.

Within this project, we updated the Information Architecture, and redesigned the navigation. For better concept communication, we created prototypes in Figma and HTML & CSS and developed a set of components ready to be shipped for development.

Alongside the set of components, this project resulted in hi-fi mock-ups that were tested and signed off by the stakeholders. This is the first step of updating the platform. The next steps would include adding more features, the frontend part for which would be easily created by site editors in the CMS.
